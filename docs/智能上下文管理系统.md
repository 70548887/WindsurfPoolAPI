# 智能上下文管理系统 (B+C+D 深度融合方案)

## 核心问题

Claude API 对话 >= 14 条消息时 100% 触发内容政策拦截。简单截断丢失上下文，用户体验极差。需要在代理层（WindsurfPoolAPI-fork）透明处理，对 OpenClaw / Codex CLI 客户端零感知。

## 架构总览

```
                    请求流
                     |
     +---------------v----------------+
     |  Pre-flight Gate (chat.js)     |  消息数 < 阈值 -> 直通(零开销)
     +---------------+----------------+
                     | >= 阈值
     +---------------v----------------+
     |  Context Manager               |  协调三层记忆
     |  (context-manager.js)          |
     +---+--------+--------+---------+
         |        |        |
    +----v---+ +--v----+ +-v----------+
    | 短期   | | 工作  | | 长期记忆   |
    | 记忆   | | 记忆  | | (SQLite)   |
    | 最近N轮| | 当前  | | 摘要+决策  |
    | 原文   | | 任务  | | +代码变更  |
    +--------+ +-------+ +------------+
         |        |        |
     +---v--------v--------v----------+
     |  Message Assembler             |  组装最终 messages[]
     |  system + 长期摘要 + 工作上下文 |
     |  + 最近原文 (总计 <= 10条)      |
     +---------------+----------------+
                     |
                     v 转发给 Cascade
```

```
                    响应流 (异步)
                     |
     +---------------v----------------+
     |  Post-response Hook (chat.js)  |  响应完成后触发
     +---------------+----------------+

     +---------------v----------------+

     +--------------------------------+
```

## 三层记忆架构 (D 深度设计)

### 层级定义

| 层级 | 数据内容 | 存储位置 | 生命周期 | 大小限制 |
|------|---------|---------|---------|---------|
| 短期记忆 | 最近 N 轮完整对话原文 | 内存 (Map) | 会话内 + SQLite 持久化 | 最近 5 轮 (10 条) |
| 工作记忆 | 当前任务目标、活跃文件、未完成操作 | 内存 (实时提取) | 每次请求重新提取 | 1 条 system 消息 |
| 长期记忆 | 滚动摘要 (任务目标/决策/代码变更/约束) | SQLite | 持久化，按 TTL 淘汰 | 每会话 1 条摘要 |

### 会话识别策略

当前系统的 `conversation-pool.js` 已有消息指纹机制，但粒度太细（全量消息哈希）。改用**前缀指纹**作为会话 ID：

```javascript
// 会话 ID = hash(前3条消息) — 同一对话的前缀稳定
function deriveConversationId(messages) {
  const anchor = messages.slice(0, 3).map(m => ({
    role: m.role,
    content: typeof m.content === 'string'
      ? m.content.slice(0, 200)
      : JSON.stringify(m.content).slice(0, 200),
  }));
  return createHash('sha256').update(JSON.stringify(anchor)).digest('hex').slice(0, 16);
}
```

## B 规则压缩 (深度优化)

### 消息重要性评分系统

不再简单按角色分类，而是为每条消息计算加权重要性分数：

```javascript
function scoreMessage(msg, index, totalCount, toolCallGraph) {
  let score = 0;

  // 1. 角色基础权重
  const roleWeight = { system: 100, tool: 70, user: 50, assistant: 30 };
  score += roleWeight[msg.role] || 20;

  // 2. 位置衰减 (越新越重要，指数衰减)
  const recency = index / totalCount;  // 0=最旧, 1=最新
  score += Math.pow(recency, 0.5) * 50;  // sqrt 衰减

  // 3. 工具调用链完整性 (被后续引用的 tool_call 不可删)
  if (msg.role === 'assistant' && msg.tool_calls?.length) {
    const isReferenced = toolCallGraph.has(msg.tool_calls[0].id);
    if (isReferenced) score += 80;  // 被引用则大幅提权
  }
  if (msg.role === 'tool') {
    score += 60;  // tool_result 始终重要
  }

  // 4. 内容信息密度 (短消息通常是确认/简单回复，长消息含关键信息)
  const contentLen = typeof msg.content === 'string' ? msg.content.length : 200;
  if (contentLen < 20) score -= 10;    // "好的"、"继续" 等低信息量
  if (contentLen > 2000) score += 10;  // 长回复通常含关键实现

  // 5. 关键词提权 (含代码/文件路径/错误信息的消息更重要)
  if (typeof msg.content === 'string') {
    if (/```[\s\S]*```/.test(msg.content)) score += 15;         // 代码块
    if (/\/[\w/]+\.\w+/.test(msg.content)) score += 10;         // 文件路径
    if (/error|bug|fix|修复|问题/i.test(msg.content)) score += 10; // 错误相关
  }

  return score;
}
```

### 智能裁剪算法

```javascript
function structuralTrim(messages, targetCount) {
  // 1. 构建工具调用依赖图
  const toolCallGraph = buildToolCallGraph(messages);

  // 2. 计算每条消息的重要性分数
  const scored = messages.map((msg, i) => ({
    msg, index: i,
    score: scoreMessage(msg, i, messages.length, toolCallGraph),
  }));

  // 3. 分离不可裁剪的消息
  const mandatory = scored.filter(s =>
    s.msg.role === 'system' ||  // system 消息永不删
    s.index >= messages.length - 6  // 最近3轮永不删
  );

  const candidates = scored.filter(s => !mandatory.includes(s));

  // 4. 按分数降序排列，保留 top-K 填满 targetCount
  candidates.sort((a, b) => b.score - a.score);
  const keepCount = targetCount - mandatory.length;
  const kept = candidates.slice(0, Math.max(0, keepCount));

  // 5. 被裁剪的消息返回给摘要生成器
  const trimmed = candidates.slice(keepCount);

  // 6. 按原始顺序重组
  const result = [...mandatory, ...kept].sort((a, b) => a.index - b.index);

  return {
    kept: result.map(s => s.msg),
    trimmedForSummary: trimmed.sort((a, b) => a.index - b.index).map(s => s.msg),
  };
}
```

## C 摘要替换 (混合模式深度优化)

### 滚动预生成 + 请求时增量更新

```
响应完成 ──> [异步] 更新滚动摘要 (覆盖到当前轮)
                              |
                              v  写入 SQLite
下次请求 ──> 检查: 新消息数 - 已覆盖轮数 = delta
             |
             delta = 0 ──> 直接用缓存摘要 (0ms)
             delta <= 2 ──> 增量合并 (快速, ~1s)
             delta > 2  ──> 完整重新生成 (~2-3s)
```

## 完整请求流程 (组装逻辑)

```javascript
// context-manager.js — 核心入口
async function processContext(messages, conversationId) {
  const totalMsgs = messages.filter(m => m.role !== 'system').length;

  // Gate: 不需要修剪
  if (totalMsgs < config.contextTrimThreshold) {
    return { messages, trimmed: false };
  }

  // 1. 从 SQLite 加载长期记忆
  const longTermMemory = db.get(
    'SELECT summary, covered_turns FROM conversation_memory WHERE conversation_id = ?',
    conversationId
  );

  // 2. 规则压缩 (B) — 分离可裁剪消息
  const { kept, trimmedForSummary } = structuralTrim(messages, 10);

  // 3. 提取工作记忆 — 从最近消息中提取当前任务上下文
  const workingMemory = extractWorkingMemory(kept);

  // 4. 摘要处理 (C 混合模式)
  let summaryText;
  if (longTermMemory && trimmedForSummary.length <= 2) {
    // 增量合并 (快速路径)
    summaryText = await mergeSummary(longTermMemory.summary, trimmedForSummary);
  } else if (longTermMemory && trimmedForSummary.length === 0) {
    // 直接复用缓存 (零延迟路径)
    summaryText = longTermMemory.summary;
  } else {
    // 完整生成 (首次或大量新消息)
    summaryText = await generateFullSummary(trimmedForSummary);
  }

  // 5. 组装最终消息数组
  const systemMsgs = messages.filter(m => m.role === 'system');
  const assembled = [
    ...systemMsgs,
    // 长期记忆注入
    ...(summaryText ? [{
      role: 'system',
      content: `[Conversation Memory]\n${summaryText}`
    }] : []),
    // 工作记忆注入
    ...(workingMemory ? [{
      role: 'system',
      content: `[Current Task Context]\n${workingMemory}`
    }] : []),
    // 短期记忆 (最近对话原文)
    ...kept.filter(m => m.role !== 'system'),
  ];

  // 6. 记录统计
  recordTrimStats(conversationId, messages.length, assembled.length, summaryText ? 'hybrid' : 'structural');

  return { messages: assembled, trimmed: true, strategy: summaryText ? 'hybrid' : 'structural' };
}
```

### 工作记忆提取 (零 API 调用)

```javascript
function extractWorkingMemory(recentMessages) {
  const parts = [];

  for (const msg of recentMessages) {
    if (msg.role !== 'user') continue;
    const content = typeof msg.content === 'string' ? msg.content : '';
    // 提取文件路径引用
    const files = content.match(/\/[\w\-./]+\.\w+/g);
    if (files) parts.push(`Active files: ${[...new Set(files)].join(', ')}`);
    // 提取任务关键词
    if (/修改|修复|添加|实现|创建|删除|重构/i.test(content)) {
      parts.push(`Current task: ${content.slice(0, 100)}`);
    }
  }

  return parts.length ? parts.join('\n') : null;
}
```
